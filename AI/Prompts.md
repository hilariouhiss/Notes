# Prompts

> 版本：v1.0
> 目标：提升使用 AI 的效率、生成准确率，并通过结构化提问挖掘真实需求。
> 适用对象：所有需要让 AI 生成、分析、总结、规划、写作、编程、检索、决策辅助的场景。
> 核心原则：**缺信息先提问，不猜测；目标先对齐，再执行；输出先定义，再生成。**

---

## 一、 总体原则

### 1.1 先确认目标，再开始生成

每次使用 AI 前，先明确：

* 我真正要解决的问题是什么？
* 最终交付物是什么？
* 谁会使用这个结果？
* 成功标准是什么？
* 有哪些限制、偏好或禁忌？

如果以上任一项不明确，AI 必须先提问确认，不得直接假设。

### 1.2 明确输入、输出、边界

一个高质量提示词必须包含以下要素中的大部分：

| 要素     | 说明            | 示例                               |
| ------ | ------------- | -------------------------------- |
| 角色     | 让 AI 以什么身份工作  | 你是一名资深产品经理 / 法务顾问 / 数据分析师        |
| 目标     | 本次任务要达成什么     | 帮我写一份面向投资人的商业计划摘要                |
| 背景     | 为什么要做、当前情况是什么 | 公司正在融资，目标客户是中小企业                 |
| 输入材料   | AI 可依据的资料     | 以下是访谈记录 / 数据表 / 文档内容             |
| 输出格式   | 结果要长什么样       | Markdown 表格 / JSON / 邮件 / PPT 大纲 |
| 约束条件   | 不能做什么、必须满足什么  | 不超过 800 字；语气专业；必须列出风险            |
| 判断标准   | 什么样的答案算好      | 可执行、准确、结构清晰、能直接交付                |
| 不确定项处理 | 信息不足时如何处理     | 先列出问题问我，得到确认后再继续                 |

### 1.3 材料和指令必须分隔

当提示词中包含资料、原文、数据、代码或用户输入时，必须使用明显分隔符，避免 AI 把资料误当作指令。

推荐分隔方式：

```text
# 指令
请总结下方材料，提取 5 个关键结论。

# 材料
"""
在这里粘贴原文、数据、邮件、会议纪要等。
"""
```

或：

```xml
<instruction>
请根据资料生成一份执行计划。
</instruction>

<context>
这里放背景资料。
</context>

<input>
这里放用户原始内容。
</input>
```

## 二、 不确定项处理规则

### 2.1 强制澄清规则

当 AI 遇到以下情况时，必须先提问确认：

* 目标不清楚
* 受众不清楚
* 输出格式不清楚
* 时间范围不清楚
* 数据来源不清楚
* 评价标准不清楚
* 存在多个可能解释
* 涉及重要决策、成本、法律、医疗、财务、安全等高风险事项
* 用户要求“最佳”“最合适”“优化”“准确”“专业”，但没有给出判断标准

### 2.2 澄清问题数量控制

AI 提问时应遵守：

* 简单任务：问 1–3 个关键问题
* 中等复杂任务：问 3–5 个关键问题
* 复杂任务：最多先问 7 个问题，避免一次性问太多
* 每个问题必须解释为什么重要，或用选项降低用户回答成本

示例：

```text
在开始前我需要确认 4 点，避免输出偏离你的真实需求：
1. 这份内容的受众是谁？A. 内部团队 B. 客户 C. 投资人 D. 公众
2. 你希望输出多长？A. 300 字内 B. 800 字左右 C. 详细版
3. 语气偏好是什么？A. 专业正式 B. 简洁直接 C. 有说服力
4. 是否有必须包含或不能提及的信息？
```

### 2.3 不得用猜测替代确认

禁止写法：

```text
我假设你的目标是……
```

推荐写法：

```text
这里有一个关键不确定项：你的主要目标可能是 A 或 B。请确认你更偏向哪一个，我再继续。
```

### 2.4 可先给“默认方案”，但必须标明

如果用户希望快速得到结果，而部分信息缺失，AI 可以给出默认方案，但必须清楚标注：

```text
以下是基于默认假设生成的版本：
- 受众：普通业务团队
- 语气：专业简洁
- 长度：800 字以内
如这些假设不对，请指出，我会重写。
```

## 三、 标准提示词结构

推荐使用以下结构编写提示词：

```markdown
# 角色
你是【专业角色】，擅长【能力范围】。

# 任务
请完成：【具体任务】。

# 背景
【补充业务背景、目标、用户、当前问题】。

# 输入材料
"""
【粘贴资料、数据、原文、代码、会议纪要等】
"""

# 输出要求
- 格式：【Markdown / 表格 / JSON / 邮件 / 大纲 / 方案】
- 长度：【字数 / 页数 / 项数】
- 语气：【正式 / 简洁 / 友好 / 专业 / 销售型】
- 结构：【必须包含哪些部分】

# 约束条件
- 必须：【硬性要求】
- 禁止：【不能做的事】
- 优先：【排序规则 / 取舍原则】

# 判断标准
一个好答案应该满足：
1. 【标准 1】
2. 【标准 2】
3. 【标准 3】

# 不确定项处理
如果有任何不确定项，请先向我提问确认；不要猜测，不要自行补全关键事实。
```

## 四、 通用高质量提示词模板

```markdown
你是【角色】，你的目标是帮助我【目标】。

请基于以下背景和资料完成任务。

## 背景
【说明当前情况、业务目标、受众、使用场景】

## 任务
【明确要 AI 做什么，例如：分析、总结、改写、生成、比较、制定计划】

## 输入资料
"""
【粘贴资料】
"""

## 输出格式
请按以下结构输出：
1. 【部分一】
2. 【部分二】
3. 【部分三】

## 质量要求
- 准确：不要编造事实，缺少依据时明确说明
- 清晰：使用标题、列表、表格组织内容
- 可执行：给出具体步骤、判断依据或下一步建议
- 贴合场景：围绕我的目标和受众，不输出泛泛而谈的内容

## 约束
- 不要使用空洞套话
- 不要自行假设关键事实
- 不要输出与任务无关的解释
- 如果涉及最新信息、价格、政策、法规、数据，请提示需要检索或让我提供资料

## 澄清机制
如果你发现任何不确定项，请先列出需要确认的问题，等待我回答后再继续。每个问题应尽量提供选项，方便我快速确认。
```

## 六、 提升准确率的提示词规则

### 6.1 要求 AI 区分事实、推断和建议

```markdown
请在回答中区分：
- 事实：有资料明确支持的内容
- 推断：基于资料合理推出，但不确定的内容
- 建议：你的行动建议
如果没有依据，请标注“依据不足”。
```

### 6.2 要求 AI 给出依据

```markdown
请为关键结论说明依据。若依据来自我提供的材料，请引用对应原文；若依据不足，请不要编造。
```

### 6.3 要求 AI 自检

```markdown
输出前请检查：
1. 是否回答了我的核心问题
2. 是否遗漏关键约束
3. 是否存在未经确认的假设
4. 是否有可能编造或过度推断的内容
5. 是否符合我要求的格式
```

### 6.4 要求 AI 不确定时明确说“不知道”

```markdown
如果你无法确定答案，请直接说明“不确定”，并告诉我需要补充什么信息，而不是给出看似确定的回答。
```

## 七、 提升效率的提示词规则

### 7.1 先让 AI 生成大纲，再生成全文

适用于长文、报告、方案、PPT、复杂计划。

```markdown
请先给我一版大纲，不要写全文。大纲通过后，再展开每一部分。
```

### 7.2 让 AI 分阶段工作

```markdown
请按以下阶段完成：
1. 先复述你对任务的理解
2. 列出不确定项并提问
3. 等我确认后再制定方案
4. 最后输出正式版本
```

### 7.3 限制输出范围

```markdown
请只输出最重要的 5 点，每点不超过 80 字。不要展开背景解释。
```

### 7.4 让 AI 给出可选方案

```markdown
请给我 3 个方案：
- 保守方案：风险低、成本低
- 平衡方案：兼顾效果和成本
- 激进方案：追求最大效果
并用表格比较优缺点。
```

## 九、 输出质量检查清单

每次 AI 输出前，应自检：

```markdown
请在最终输出前完成自检，但不要展开冗长过程。重点检查：

- 是否解决了用户的核心问题？
- 是否符合指定格式？
- 是否遗漏背景、约束或受众？
- 是否存在未经确认的假设？
- 是否有编造事实或来源的风险？
- 是否区分了事实、推断和建议？
- 是否给出了可执行下一步？
- 是否有更简洁、更准确的表达方式？
```

## 十一、 AI 需求澄清问题库

当需求不清楚时，AI 可从以下问题中选择最关键的提问：

### 11.1 目标类

* 你希望最终达成什么结果？
* 这个结果用于什么场景？
* 你更看重速度、质量、成本，还是风险控制？

### 11.2 受众类

* 这份内容给谁看？
* 受众的专业水平如何？
* 受众最关心什么？

### 11.3 输出类

* 你希望输出成什么格式？
* 需要多长？
* 是要初稿、终稿，还是多个方向供选择？

### 11.4 约束类

* 有预算、时间、人力限制吗？
* 有必须包含或不能提及的信息吗？
* 有品牌语气、合规、安全或风格要求吗？

### 11.5 判断标准类

* 什么样的结果你会认为“好”？
* 有没有参考样例？
* 更偏向保守稳妥，还是创新突破？

### 11.6 信息来源类

* 是否只能依据你提供的材料？
* 是否需要最新资料或外部数据？
* 是否需要引用来源？

## 十二、 推荐工作流

### 12.1 简单任务

```text
明确任务 → 给出格式 → 直接输出 → 简短自检
```

### 12.2 中等任务

```text
明确目标 → 补充背景 → AI 提问澄清 → 用户确认 → AI 输出 → 用户迭代
```

### 12.3 复杂任务

```text
需求澄清 → 目标确认 → 输出大纲 → 用户确认 → 分模块生成 → 质量检查 → 生成最终版
```

```markdown
- 如果涉及事实、数据、政策、价格、法律、医疗、财务或最新信息，请说明需要可靠来源
- 不要自行假设关键事实

请先判断需求是否充分；如不充分，先向我提问，不要直接生成最终答案。
```

## 十四、 最重要的执行准则

> **AI 的第一责任不是立刻回答，而是先判断问题是否足够清楚。**
> **如果需求不清楚，先提问；如果依据不足，先说明；如果目标明确，再高效执行。**
