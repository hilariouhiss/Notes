# AI辅助编码规范：嵌入式与汽车电子行业版

适用对象：嵌入式软件开发团队、汽车电子 ECU/域控/中央计算平台团队、测试团队、功能安全团队、网络安全团队、架构师、技术负责人、AI辅助开发使用者

> 说明：本文档为嵌入式与汽车电子行业的工程规范模板，默认以“必须/应该/可以”区分约束强度。实际落地时，应结合 OEM 客户要求、项目 ASIL 等级、Cybersecurity Impact、AUTOSAR/非 AUTOSAR 架构、芯片平台、工具链、量产计划、组织流程成熟度和适用法规进行裁剪。本文档不替代 ISO 26262、ISO/SAE 21434、Automotive SPICE、AUTOSAR、MISRA 等标准原文，也不构成认证结论。

## 二、目的、范围与术语

### 2.1 目的

- 建立适用于嵌入式与汽车电子软件的统一工程规范，降低 AI 辅助编码在安全、实时性、资源、合规、可维护性和量产交付中的风险。
- 确保 AI 生成或 AI 改写的代码与人工编写代码遵循同等质量要求，并满足可追溯、可验证、可审计、可复现的工程要求。
- 将需求、设计、开发、测试、评审、发布、量产、售后和持续改进闭环化，提高 ECU、域控制器、中央计算平台、网关、传感器控制器和执行器控制软件的交付稳定性。
- 支持与功能安全、网络安全、ASPICE、AUTOSAR、MISRA、诊断、OTA、制造和售后流程的协同。

### 2.2 适用范围

- 适用于 MCU、MPU、SoC、ECU、域控制器、中央计算平台、智能传感器、执行器控制器、网关、车身控制、动力/底盘/热管理/座舱/智驾相关嵌入式软件。
- 适用于 AUTOSAR Classic、AUTOSAR Adaptive、OSEK/VDX、RTOS、裸机、Linux/QNX/Android Automotive 等平台上的量产软件、工具脚本、测试软件和生成代码。
- 适用于 AI 参与的活动：需求澄清、架构设计、接口设计、代码生成、代码解释、重构、测试生成、静态分析解释、缺陷定位、文档生成、诊断策略设计、脚本生成和配置辅助。
- 不适用于以 AI 自主决策直接控制车辆的在线运行时模型规范；如涉及感知、规划、控制或自动驾驶算法，应另行纳入 SOTIF、数据闭环、模型验证和场景库规范。

### 2.3 术语定义

| 术语 | 定义 |
|---|---|

| SWC | Software Component，软件组件，常用于 AUTOSAR 应用层组件。 |

## 三、参考标准与裁剪原则

### 3.1 参考标准与框架

项目应根据客户要求、市场区域、车型平台、功能安全等级和网络安全影响，选择适用标准并形成项目级裁剪说明。

| 类别 | 常见参考 |
|---|---|
| 功能安全 | ISO 26262:2018 系列、项目安全计划、HARA、FSC/TSC、软件安全需求、Safety Case。 |
| 预期功能安全 | ISO 21448:2022，适用于依赖复杂传感器、环境感知、自动驾驶或辅助驾驶功能的场景。 |
| 汽车网络安全 | ISO/SAE 21434:2021、TARA、Cybersecurity Case、漏洞管理、事件响应。 |
| 法规与型式认证 | UNECE R155、R156，以及目标市场对应法规和 OEM 要求。 |
| 软件更新 | ISO 24089、SUMS、OTA 更新包、版本管理、回滚和更新记录。 |
| 过程能力 | Automotive SPICE 4.0、客户过程要求、内部 QMS。 |
| 架构与平台 | AUTOSAR Classic、AUTOSAR Adaptive、OSEK/VDX、POSIX、RTOS、Linux/QNX/Android Automotive。 |
| 编码规范 | MISRA C:2023、MISRA C++:2023、AUTOSAR C++14、CERT C/C++、项目编译器安全子集。 |
| 诊断与通信 | UDS、DoIP、CAN/CAN FD、LIN、FlexRay、Automotive Ethernet、SOME/IP、DDS、车载网络管理规范。 |
| 硬件与质量 | AEC-Q、芯片供应商安全手册、FMEDA、硬件安全机制说明、编译器/工具链合格性资料。 |

### 3.2 裁剪原则

- 必须建立项目级《适用标准与裁剪矩阵》，明确每个标准或客户条款的适用性、裁剪理由、责任人和证据位置。
- 必须将 ASIL、Cybersecurity Impact、车辆功能风险、量产阶段和客户交付物要求映射到工程活动强度。
- 对安全相关软件，必须以更严格要求优先；当客户规范、法规、功能安全、网络安全和内部规范冲突时，应采用更严格要求或发起正式偏差审批。
- 对 AI 参与的活动，必须明确 AI 输出是否属于开发工具输出、辅助建议、SOUP 或可验证中间产物，并确定对应验证策略。
- 可以按项目成熟度分阶段落地，但不得裁剪掉安全目标、网络安全目标、关键需求追踪、关键测试、缺陷闭环和发布回滚要求。

## 四、总体原则与规范等级

### 4.1 总体原则

- 人对结果负责：AI 可以辅助，但不得替代工程判断、功能安全确认、网络安全确认、代码评审、测试结论和发布审批。
- 安全优先：便利性、开发速度和成本不得凌驾于车辆安全、网络安全、数据安全、法规合规和客户安全目标之上。
- 小步可验证：优先拆分为小需求、小设计、小提交、小 PR、小版本，避免一次性引入大段不可解释代码。
- 可追溯：需求、设计、代码、测试、缺陷、风险接受、工具结果、AI 使用记录和发布记录必须能互相追踪。
- 确定性优先：嵌入式软件应优先选择确定性时序、确定性内存、确定性错误处理和可复现构建。
- 防御式设计：任何外部输入、总线消息、诊断请求、传感器数据、标定参数、NVM 数据、OTA 包和 AI 输出都不得默认可信。
- 最小权限与最小暴露：只向 AI 工具提供完成任务所需的最少上下文；未经批准不得上传完整代码库、ARXML、DBC、客户需求、密钥、日志或车辆数据。

### 4.2 规范等级

| 等级 | 含义 |
|---|---|
| 必须 | 强制要求。违反后不得合并、发布、交付或上车，除非获得正式书面豁免。 |
| 应该 | 推荐要求。不采用时应在设计文档、PR 或偏差记录中说明原因。 |
| 可以 | 可选实践。由团队根据场景、成本和收益选择。 |

## 六、AI辅助编码使用规范

### 6.1 允许使用场景

- 需求摘要、边界条件梳理、验收标准草拟、测试点扩展。
- 方案对比、接口草案、状态机草案、错误处理清单、异常路径梳理。
- 非安全关键样板代码、脚手架、局部函数、测试用例、Mock/Fake、工具脚本和文档草稿。
- 代码解释、重构建议、静态分析结果解释、单元测试补充、缺陷定位假设。
- CAN/LIN/Ethernet 信号处理样例、UDS 服务处理框架、NVM 参数校验逻辑草案，但必须由人工对照协议和项目配置验证。
- 配置示例、构建脚本、CI 脚本、解析脚本，但必须在隔离环境验证，不得直接操作生产或量产基线。

### 6.2 禁止或受限场景

- 禁止向未经批准的 AI 工具输入密钥、证书、令牌、完整客户需求、完整源代码库、未公开漏洞、量产车辆日志、VIN、个人信息、商业机密、ARXML/DBC/ODX/PDX 等受限资料。
- 禁止让 AI 直接修改生产环境、车辆、HIL 台架、发布仓库、刷写包、诊断数据库、密钥系统或量产配置。
- 禁止在未理解、未测试、未评审的情况下合入 AI 生成代码。
- 禁止让 AI 生成绕过安全机制、禁用看门狗、关闭诊断、屏蔽 DTC、降低安全校验或绕过 Secure Boot/OTA 验签的实现。
- 涉及制动、转向、动力、气囊、高压电、热失控、自动驾驶、远程控制、身份认证、密钥管理、诊断解锁、Bootloader、OTA 的代码，AI 输出只能作为参考，必须进行专项评审。
- 对 ASIL B 及以上或 Cybersecurity Impact 高的模块，AI 生成代码不得作为“黑盒贡献”进入基线；必须补充设计说明、逐行审查、测试证据和风险评估。

### 6.3 AI 使用四步法

| 步骤 | 要求 |
|---|---|
| 1. 定界 | 明确目标、输入输出、平台、编译器、RTOS/AUTOSAR 版本、资源限制、ASIL/网络安全等级、不可泄露信息和验收标准。 |
| 2. 生成 | 要求 AI 输出方案、假设、边界、代码、测试建议、潜在风险和待确认问题；禁止只要代码不问假设。 |
| 3. 验证 | 人工审查、协议核对、静态分析、编译、单元测试、SIL/PIL/HIL、覆盖率、安全扫描和需求追踪。 |
| 4. 记录 | 在任务、PR 或评审记录中说明 AI 使用环节、输入是否脱敏、采纳内容、人工修改点、验证结果和剩余风险。 |

### 6.4 提示词要求

- 提示词必须描述平台约束：芯片、编译器、语言标准、RTOS/AUTOSAR 层级、无动态内存/无递归/无异常等项目规则。
- 需要 AI 生成代码时，应要求其先给出设计思路、状态机、边界条件、错误路径和测试点，再输出代码。
- 涉及已有代码时，只提供最小必要上下文；敏感信息、客户标识、车辆数据和内部路径必须脱敏。
- 应要求 AI 标注不确定点，不得诱导 AI 编造不存在的 API、寄存器、诊断服务、标定项、ARXML 配置或芯片特性。
- 对协议、寄存器、AUTOSAR API、编译器扩展、RTOS API 和标准库行为，必须对照项目文档、供应商手册或官方资料验证。

## 七、需求工程与可追溯性

### 7.1 需求澄清

- 必须明确车辆场景、用户操作、环境条件、输入信号、输出动作、状态机、权限、异常场景、降级策略和验收标准。
- 必须识别非功能需求：启动时间、周期时间、端到端延迟、CPU 负载、RAM/ROM、NVM 寿命、功耗、诊断可用性、功能安全、网络安全、可刷写性、售后可诊断性。
- 涉及安全相关功能时，必须区分系统需求、软件需求、软件安全需求和软件单元需求。
- 涉及网络安全相关功能时，必须识别资产、攻击面、威胁、网络安全目标、网络安全需求和验证方法。
- 可以使用 AI 辅助生成问题清单和边界条件，但业务事实、客户要求、法规要求和安全目标必须由人工确认。

### 7.2 可追溯性

- 必须建立从客户需求/系统需求到软件需求、架构、详细设计、代码、单元测试、集成测试、系统测试、缺陷和发布版本的双向追踪。
- 安全需求和网络安全需求必须具有独立标识、ASIL/影响等级、验证方法、评审状态和证据链接。
- 所有 AI 生成或 AI 修改的重要逻辑必须能追踪到需求、设计说明、测试用例和评审记录。
- 需求变更必须触发影响分析，包括架构、接口、测试、标定、诊断、网络、NVM、资源和发布影响。
- 不得以“AI 已确认”“AI 认为没影响”作为变更影响分析结论。

### 7.3 需求质量检查

| 检查项 | 要求 |
|---|---|
| 原子性 | 一个需求只表达一个可验证意图。 |
| 可验证性 | 必须有明确测试方法或分析方法。 |
| 无歧义 | 避免“快速”“可靠”“尽量”“适当”等不可测表述。 |
| 边界条件 | 必须明确无效值、丢帧、超时、掉电、复位、诊断会话、休眠唤醒等场景。 |
| 安全/网络安全属性 | 必须标注 ASIL、Cybersecurity Impact 或项目定义的等级。 |
| 追踪状态 | 必须能追踪到上游需求和下游验证证据。 |

## 八、功能安全与安全生命周期

### 8.1 基本要求

- 安全相关项目必须建立安全计划，明确安全活动、工作产品、里程碑、确认措施、责任人和独立性要求。
- 必须根据 HARA、ASIL 分解、FSC/TSC 和软件安全需求确定软件开发强度。
- 软件安全需求必须被分配到架构元素、软件组件、软件单元和测试用例。
- 安全机制必须设计监控对象、检测条件、故障反应时间、诊断覆盖率目标、故障存储、降级状态和恢复策略。
- 安全相关代码必须通过设计评审、静态分析、单元测试、集成测试、故障注入和安全验证。
- AI 不得替代安全分析、ASIL 判断、安全确认评审或 Safety Case 论证。

### 8.2 软件安全设计

- 必须明确安全状态和安全降级状态，例如关闭输出、限制扭矩、进入跛行模式、保持当前状态、请求驾驶员接管或触发系统复位。
- 必须识别潜在单点故障、潜伏故障、共因失效和相关失效。
- 对 ASIL 相关数据，必须定义范围检查、合理性检查、时序检查、E2E 保护、CRC、计数器、超时、冗余、反演存储或其他安全机制。
- 安全机制应尽量独立于被监控功能，避免同源错误同时破坏功能和监控。
- 安全相关通信必须考虑丢帧、重复帧、错序、超时、篡改、错误长度、非法值和总线关闭。
- 必须定义看门狗策略，包括触发条件、喂狗位置、任务监督、窗口看门狗、复位后的故障记录和恢复策略。

### 8.3 安全工作产品

| 工作产品 | 内容要求 |
|---|---|
| 安全计划 | 活动、里程碑、角色、独立性、确认措施和证据清单。 |
| HARA 输入/输出 | 场景、危害、严重度、暴露率、可控性、ASIL 和安全目标。 |
| FSC/TSC | 功能安全概念、技术安全概念、安全机制和系统分配。 |
| 软件安全需求 | 软件层面的安全要求、ASIL、验证方法、追踪关系。 |
| 安全架构 | 安全机制、分区、监控、故障反应、接口和资源预算。 |
| 安全测试证据 | 单元、集成、故障注入、边界、时序、覆盖率和 HIL 结果。 |
| Safety Case | 安全论证、证据索引、偏差和剩余风险接受。 |

## 九、汽车网络安全与软件更新

### 9.1 网络安全工程

- 必须识别资产、攻击面、信任边界、威胁场景、攻击路径、风险等级和网络安全目标。
- 必须将网络安全需求分配到系统、软件、硬件、制造、运维、售后和供应链环节。
- 必须对诊断、远程服务、蓝牙/Wi-Fi/蜂窝/V2X、以太网、USB、OTA、调试口、日志接口、云端接口和供应商工具进行攻击面分析。
- 必须对身份认证、访问控制、密钥管理、加密、完整性、重放保护、安全日志和事件响应进行设计评审。
- 诊断解锁、刷写、安全启动、OTA、证书更新和密钥派生不得使用 AI 直接生成的未验证实现。
- 不得将密钥、证书、种子密钥算法、调试口密码、生产解锁机制或安全事件细节输入未经批准的 AI 工具。

### 9.2 软件更新与 OTA

- 必须定义软件更新管理流程，包括版本识别、兼容性、依赖关系、签名、完整性校验、更新前检查、失败恢复、回滚、记录和售后追踪。
- OTA 包必须具备身份认证、签名验证、完整性校验、防回滚策略、断点恢复策略和更新状态机。
- 更新过程必须考虑断电、通信中断、空间不足、版本不兼容、依赖缺失、双分区切换失败和回滚失败。
- 更新日志必须可追踪到车辆、ECU、软件版本、配置版本、标定版本、时间、结果和失败原因；涉及个人信息时必须遵守隐私要求。
- AI 可以辅助生成更新状态机草案和测试用例，但状态机必须由架构、网络安全、测试和发布负责人联合确认。

### 9.3 漏洞管理

- 必须建立第三方组件、开源库、编译器、RTOS、Bootloader、通信栈、加密库和工具链的漏洞监控机制。
- 高风险漏洞必须进行影响分析、修复计划、验证、发布策略和客户沟通。
- 网络安全事件必须记录发现来源、影响范围、复现方式、攻击可行性、缓解措施、永久修复和证据。
- AI 辅助漏洞分析时，必须验证其假设；不得直接执行 AI 建议的高风险命令或绕过安全控制。

## 十、软件架构与模块边界

### 10.1 架构原则

- 必须明确软件分层、组件边界、接口契约、数据流、控制流、故障传播路径、资源预算和时序预算。
- 安全相关软件必须支持分区、隔离、监控和降级；高 ASIL 组件不得随意依赖低完整性组件。
- 模块之间应通过明确接口、事件、RTE Port、服务接口或消息契约交互，禁止随意访问其他模块内部数据。
- 公共组件必须保持通用性，避免掺入特定车型、客户或项目硬编码假设。
- 同一功能的状态机必须集中定义，避免多处隐式状态和重复判断导致不一致。
- 架构必须考虑上电、掉电、复位、休眠、唤醒、刷写、诊断会话、总线关闭、故障恢复和版本升级。

### 10.2 分层约束

| 层次 | 职责边界 |
|---|---|
| 应用层/功能层 | 车辆功能逻辑、状态机、控制策略、诊断业务语义、故障反应。 |
| 服务层/平台抽象 | NVM、通信、诊断、时间、CRC、Watchdog、Crypto、存储和系统服务封装。 |
| BSW/OS/驱动层 | MCAL、RTOS、调度、通信栈、诊断栈、存储栈、硬件驱动和中断。 |
| 硬件抽象层 | 寄存器、外设、芯片差异、板级配置和硬件安全机制抽象。 |
| 工具/配置层 | ARXML、DBC、ODX、PDX、A2L、链接脚本、生成代码和构建配置。 |

### 10.3 接口契约

- 接口必须定义数据类型、单位、范围、初始值、无效值、更新时间、超时策略、端序、缩放因子和错误语义。
- 总线信号必须明确周期、发送条件、接收超时、初始值、丢帧处理、错误值、诊断关联和安全保护机制。
- 内部接口不得传播未经校验的外部输入；必须在边界层完成基础校验和可信度标注。
- 任何接口变更必须触发影响分析，包括调用方、测试、标定、诊断、网络矩阵、NVM、版本兼容和售后工具。

## 十一、AUTOSAR 与平台软件规范

### 11.1 AUTOSAR Classic

- 必须明确 SWC、RTE、BSW、MCAL、CDD、Complex Driver、OS Task、Event、Runnable 和资源映射关系。
- ARXML 配置变更必须进入版本控制，并能追踪到需求、网络矩阵、诊断规范和测试结果。
- RTE 接口必须稳定，变更必须进行调用方影响分析和集成测试。
- Runnable 必须明确触发方式、周期、最坏执行时间、数据一致性、重入性和错误处理。
- BSW 配置必须经过评审，重点检查 COM、PduR、CanIf、CanSM、Nm、Dem、Dcm、NvM、WdgM、Crypto、E2E、OS 和 Memory Mapping。
- Complex Driver 必须限制范围，说明绕过标准层的原因、接口、风险和测试证据。

### 11.2 AUTOSAR Adaptive

- 必须明确服务接口、Execution Management、Communication Management、Persistency、Diagnostics、Security、Logging、Update and Configuration 的使用边界。
- Adaptive 应用必须定义进程生命周期、健康监控、服务发现、订阅恢复、故障隔离、持久化策略和更新策略。
- POSIX/Linux/QNX 等环境中的多进程、多线程、文件系统、网络栈和权限必须进行安全设计。
- 服务接口变更必须考虑向后兼容、版本协商、订阅方影响、回滚和灰度策略。

### 11.3 非 AUTOSAR 平台

- 必须定义平台抽象层，隔离芯片、外设、RTOS、编译器和板级差异。
- 裸机或 RTOS 项目必须定义任务模型、优先级、中断模型、共享资源、时间基准、初始化顺序和故障恢复策略。
- 不得在业务逻辑中直接散布寄存器访问、编译器扩展、魔法地址或板级宏；必须通过受控接口封装。

## 十二、硬件抽象、驱动与寄存器访问

### 12.1 驱动开发

- 驱动必须明确硬件资源、寄存器、时钟、复位、引脚、DMA、中断、缓存一致性、电源状态和错误状态。
- 寄存器访问必须使用受控宏、结构体或供应商推荐方式；禁止在业务代码中使用未封装的魔法地址。
- 访问硬件寄存器必须使用合适的 `volatile` 或内存屏障；不得滥用 `volatile` 替代并发保护。
- 驱动接口必须定义初始化前、初始化中、运行中、休眠、唤醒、错误和反初始化状态下的行为。
- DMA、缓存、MPU/MMU、Endianness 和对齐要求必须在设计中说明并测试。
- 对安全相关驱动，必须结合芯片 Safety Manual 设计故障检测和诊断机制。

### 12.2 中断与任务交互

- ISR 必须短小、确定、无阻塞；不得在 ISR 中执行长循环、复杂业务、动态分配、文件操作、非确定性日志或可能阻塞的 API。
- ISR 与任务共享数据必须使用原子操作、临界区、锁、队列或双缓冲，并说明竞态保护策略。
- 必须定义中断优先级、嵌套策略、禁止中断时间上限和最坏响应时间。
- 禁止在多个上下文中无保护地读写同一变量；AI 生成代码必须重点检查竞态、重入和内存可见性。

## 十三、通信、诊断与网络管理

### 13.1 车载通信

- CAN/CAN FD、LIN、FlexRay、Ethernet、SOME/IP、DDS 等通信必须明确报文 ID、周期、长度、信号范围、端序、超时、初始值、错误值和版本。
- 接收路径必须校验长度、计数器、CRC、E2E、范围、时序、源地址、会话状态和安全权限。
- 发送路径必须控制总线负载、发送周期、抖动、突发、优先级和错误恢复。
- 对安全相关信号，必须定义丢帧、重复、错序、非法值、总线关闭、延迟超限和网关转换异常下的行为。
- 通信矩阵、DBC、ARXML 和代码必须一致；生成代码必须有版本和配置基线。

### 13.2 诊断规范

- UDS 服务必须定义会话、安全等级、子功能、请求长度、响应、NRC、前置条件、执行时间、DTC 关联和安全影响。
- 安全相关或网络安全相关诊断服务必须经过专项评审，如 ECU Reset、Security Access、Routine Control、Request Download、Transfer Data、Write Data By Identifier、Control DTC Setting 等。
- DTC 必须定义检测条件、成熟条件、恢复条件、老化策略、快照数据、扩展数据、清除策略和仪表/策略影响。
- 诊断例程不得绕过安全策略；生产例程、开发例程和售后例程必须隔离并通过权限控制。
- AI 可以辅助生成诊断测试点，但不得凭空定义 DID、RID、DTC、NRC 或安全访问算法。

### 13.3 网络管理与电源模式

- 必须定义上电、KL15/KL30、休眠、唤醒、部分网络、总线休眠、刷写模式、运输模式和工厂模式下的软件行为。
- 网络管理状态机必须考虑唤醒源、超时、总线错误、远程唤醒、重复唤醒和异常掉电。
- 低功耗路径必须控制外设关闭顺序、NVM 写入、通信停止、唤醒源配置和恢复校验。

## 十四、Bootloader、启动链与 OTA

### 14.1 启动链

- 必须定义启动顺序、上电自检、时钟初始化、内存初始化、Bootloader 跳转、应用校验、异常恢复和安全状态。
- 安全启动必须验证镜像完整性和真实性；验证失败必须进入受控恢复路径，不得继续运行不可信软件。
- Bootloader 与 App 的接口必须稳定，包括跳转地址、向量表、共享内存、NVM 标志、版本信息和错误码。
- 必须定义 A/B 分区或等效回滚策略，确保刷写失败、断电、校验失败时可恢复。

### 14.2 刷写与更新

- 刷写流程必须定义会话切换、安全访问、擦写、下载、传输、校验、激活、复位、失败恢复和版本记录。
- 更新包必须绑定 ECU、硬件版本、软件版本、配置版本、标定版本、依赖关系和兼容矩阵。
- 禁止在量产刷写链路中使用未评审的 AI 生成脚本或未验证的诊断序列。
- 刷写工具、脚本、ODX/PDX、密钥和证书必须纳入配置管理和访问控制。

## 十五、编码规范：C/C++/脚本/配置

### 15.1 通用要求

- 量产嵌入式 C/C++ 代码必须遵循项目指定的 MISRA、CERT、AUTOSAR C++ 或客户编码规范，并通过批准的静态分析工具检查。
- 编译器警告必须作为缺陷处理；安全相关模块应将关键警告升级为错误。
- 必须使用固定宽度整数类型或项目定义的类型别名，禁止依赖 `int`、`long` 等类型宽度假设。
- 所有类型转换必须显式、可解释、可测试；禁止隐式窄化、符号变化和未审查的指针转换。
- 禁止未定义行为、实现定义行为和未指定行为进入安全相关代码；无法避免时必须形成偏差记录和验证证据。
- 禁止提交调试代码、无用日志、临时开关、硬编码凭证、个人路径、未使用代码和无评审的生成文件。
- 公共 API、复杂算法、安全机制、诊断策略和状态机必须注释“为什么”和边界，而不是重复“做什么”。

### 15.2 C 语言要求

- 必须限制指针算术、强制转换、联合体类型穿透、宏副作用和未受控全局变量。
- 宏必须加括号，禁止多次求值副作用；优先使用 `static inline` 函数替代复杂宏。
- 数组访问必须进行边界控制；不得假设外部输入长度正确。
- 位操作必须使用命名掩码、明确位宽和类型后缀；不得使用魔法数字表达寄存器或协议字段。
- 函数必须有完整原型；内部函数应使用 `static` 限定可见性。
- 禁止递归；禁止非受控动态内存；禁止在安全关键路径使用不确定执行时间的库函数。

### 15.3 C++ 要求

- 项目必须明确 C++ 标准版本、允许/禁止特性、标准库使用范围、异常、RTTI、模板、动态分配和多线程策略。
- 安全关键嵌入式软件默认禁止运行期异常、RTTI 和非受控动态分配；确需使用时必须专项评审并提供资源和失败路径证据。
- 构造/析构不得依赖不确定初始化顺序；全局对象和静态对象初始化必须受控。
- 虚函数、继承、模板、lambda、智能指针、容器和并发库必须遵循项目规则和静态分析约束。
- RAII 可以用于资源管理，但必须验证构造失败、析构时序和实时性影响。

### 15.4 脚本与配置

- Python、Shell、CAPL、构建脚本、刷写脚本、解析脚本和生成脚本必须进入版本控制并经过评审。
- 任何会影响量产代码、生成文件、刷写包、配置包或测试结论的脚本，必须有输入校验、错误处理、日志和回归测试。
- AI 生成脚本必须在隔离环境和样例数据上验证，禁止直接作用于量产仓库、发布包或车辆。
- ARXML、DBC、A2L、ODX、PDX、链接脚本、编译选项、诊断配置和标定配置必须纳入基线管理。

## 十六、实时性、并发、内存与资源管理

### 16.1 实时性

- 必须为周期任务、事件任务、ISR、通信处理、诊断服务、控制算法和安全监控定义时间预算。
- 必须测量或分析最坏执行时间、抖动、抢占、阻塞时间、总线延迟和端到端延迟。
- 禁止在实时关键路径执行未受控循环、阻塞等待、动态分配、文件访问、网络阻塞调用或高复杂度日志。
- 性能优化必须基于测量，不得仅凭 AI 建议修改调度或资源分配。

### 16.2 内存管理

- 必须定义 ROM、RAM、栈、堆、NVM、共享内存、DMA 缓冲和校准区预算。
- 安全关键软件应避免运行期动态内存；确需使用时必须定义分配时机、上限、失败行为、碎片评估和测试证据。
- 栈大小必须通过静态分析、运行监控或压力测试评估；禁止以经验值随意配置。
- NVM 写入必须考虑掉电一致性、写入次数、磨损、CRC、版本迁移、默认值和恢复策略。
- 内存映射和链接脚本变更必须评审，检查段属性、启动初始化、对齐、保留区、校验区和安全分区。

### 16.3 并发与同步

- 必须识别共享资源、访问上下文、保护机制、死锁风险、优先级反转和重入性。
- 锁、信号量、队列、事件、临界区和原子操作必须有超时或上限策略；安全关键路径不得无限等待。
- 多核系统必须考虑缓存一致性、内存屏障、核间通信、任务亲和性和共享资源隔离。
- AI 生成并发代码必须重点检查竞态、死锁、活锁、重入、内存可见性和优先级反转。

## 十七、错误处理、降级与故障管理

### 17.1 错误处理

- 所有外部输入、通信消息、诊断请求、NVM 数据、标定参数、传感器数据和工具生成数据必须校验。
- 错误处理必须明确错误码、错误等级、故障记录、重试策略、降级策略、恢复条件和用户/上层通知。
- 禁止吞掉异常或错误返回值；确需忽略时必须说明原因并限制影响范围。
- 对外错误响应不得泄露密钥、内部路径、堆栈、调试口信息或安全机制细节。

### 17.2 降级与故障反应

- 必须为关键故障定义安全状态、降级策略、故障反应时间、恢复条件和复位策略。
- 故障处理必须避免“抖动式恢复”，应考虑去抖、成熟、锁存、恢复延迟和驾驶员提示。
- DTC、故障状态、降级状态和功能输出之间必须有清晰映射。
- 故障注入测试必须覆盖传感器异常、通信超时、NVM 损坏、看门狗、任务卡死、总线关闭、掉电、复位和刷写失败。

## 十八、AI生成代码接收规范

### 18.1 接收前检查表

- [ ] 是否确认 AI 输出解决的是当前需求，而不是类似但不同的假设场景？
- [ ] 是否逐行理解关键逻辑、状态变化、资源访问、异常路径和副作用？
- [ ] 是否对照项目 API、AUTOSAR API、芯片手册、RTOS 文档、协议规范或客户文档验证？
- [ ] 是否检查未定义行为、越界、溢出、符号转换、端序、对齐、竞态、死锁和实时性风险？
- [ ] 是否检查是否引入新依赖、开源许可证风险、供应链风险或不可量产工具？
- [ ] 是否为 AI 生成逻辑补充单元测试、异常测试、边界测试和集成测试？
- [ ] 是否通过编译、静态分析、MISRA/CERT 检查、覆盖率、SIL/PIL/HIL 或必要手工验证？
- [ ] 是否在 PR 或任务记录中说明 AI 参与范围、人工修改点和验证结果？

### 18.2 AI 输出常见风险

| 风险 | 控制措施 |
|---|---|
| 编造 API/寄存器/配置 | 必须对照项目代码、供应商手册或官方文档验证。 |
| 忽略实时性 | 必须测量或分析 WCET、阻塞、抖动和资源占用。 |
| 忽略嵌入式约束 | 检查动态内存、递归、异常、标准库、浮点、printf、文件系统等使用。 |
| 隐藏安全漏洞 | 检查输入校验、权限、密钥、诊断解锁、刷写、OTA 和日志脱敏。 |
| 测试虚假通过 | 测试必须验证业务行为和安全需求，不得只验证实现细节。 |
| 许可证不明 | 大段实现必须进行来源和许可证风险评估。 |
| 上下文误解 | 需要人工确认需求、接口、项目版本、编译器、RTOS 和客户规则。 |
| 不可追溯 | 必须补充需求、设计、测试和评审证据。 |

### 18.3 高风险 AI 输出处理

- 涉及 ASIL B 及以上、Cybersecurity Impact 高、诊断安全访问、Bootloader、OTA、密钥、加密、制动/转向/动力/高压/热管理的 AI 输出，必须进行专项评审。
- 评审必须至少覆盖功能正确性、安全机制、网络安全、实时性、资源、可测试性、可追溯性、偏差和验证证据。
- 如 AI 工具参与自动生成量产代码或正式测试证据，应评估其工具置信度和工具验证策略；不得默认 AI 工具已合格。

## 十九、代码评审与变更评审

### 19.1 PR 基本要求

- PR 应保持小而聚焦；一个 PR 原则上只解决一个明确目标。
- PR 说明必须包含背景、需求链接、变更内容、影响范围、测试结果、资源影响、安全/网络安全影响、回滚方式和 AI 参与说明。
- 涉及接口、ARXML、DBC、A2L、ODX、NVM、Bootloader、诊断、密钥、调度、内存、编译选项和链接脚本的变更必须明确标注。
- 未通过 CI、编译、静态分析、单元测试或必要集成测试的 PR 不得合并。
- 高风险变更必须由架构、测试、功能安全、网络安全或平台负责人进行专项评审。

### 19.2 评审关注点

| 维度 | 检查重点 |
|---|---|
| 需求一致性 | 是否满足需求和验收标准，是否引入非预期功能。 |
| 安全一致性 | 是否满足软件安全需求，是否影响 ASIL 分解和安全机制。 |
| 网络安全 | 是否影响攻击面、认证、权限、密钥、诊断、刷写、OTA 和日志。 |
| 架构边界 | 是否符合分层、接口、状态机、资源和平台约束。 |
| 实时与资源 | 是否影响 CPU、RAM、ROM、栈、NVM、总线负载、周期和延迟。 |
| 正确性 | 边界、异常、并发、端序、对齐、溢出、超时、重入和复位路径。 |
| 测试充分性 | 单元、集成、SIL/PIL/HIL、故障注入、回归、覆盖率和诊断测试。 |
| 可维护性 | 命名、复杂度、注释、重复代码、生成代码边界和文档同步。 |
| AI 风险 | AI 输出是否已理解、验证、测试、记录，是否存在幻觉或许可证风险。 |

## 二十、测试与验证确认

### 20.1 测试分层

| 测试类型 | 目标 |
|---|---|
| 静态测试 | 编译、静态分析、MISRA/CERT/AUTOSAR 规则、复杂度、依赖和敏感信息。 |
| 单元测试 | 验证函数、类、软件单元、边界条件、错误路径和安全需求。 |
| 组件测试 | 验证 SWC、模块或服务层接口、状态机、Mock BSW/硬件交互。 |
| 集成测试 | 验证模块间、RTE、BSW、OS、通信栈、诊断栈、NVM 和外设交互。 |
| SIL | 在 PC 或仿真环境验证算法、状态机、诊断逻辑和回归。 |
| PIL | 在目标处理器或等效处理器上验证编译器、指令集、时序和数值一致性。 |
| HIL | 在台架上验证 ECU、网络、传感器/执行器、故障注入和车辆级场景。 |
| 诊断测试 | 验证 UDS 服务、DTC、快照、扩展数据、安全访问、刷写和售后流程。 |
| 性能测试 | 验证 CPU、RAM、ROM、栈、启动时间、周期、延迟、总线负载和 NVM 寿命。 |
| 安全测试 | 验证安全机制、故障反应、故障注入、降级、复位恢复和覆盖率。 |
| 网络安全测试 | 验证认证、授权、输入健壮性、模糊测试、渗透测试、OTA/诊断安全。 |

### 20.2 测试要求

- 新增或修改业务逻辑必须补充自动化测试；缺陷修复必须优先添加能复现缺陷的测试。
- 测试必须覆盖正常、异常、边界、超时、丢帧、复位、掉电、NVM 损坏、诊断会话、权限和降级场景。
- AI 生成测试必须由人工确认断言有效，防止测试跟随错误实现。
- 安全相关需求必须有独立验证证据；网络安全需求必须有安全测试或分析证据。
- HIL 测试必须记录台架配置、硬件版本、软件版本、标定版本、网络配置、测试脚本版本和环境条件。
- 对浮点、定点、查表、插值、滤波、控制算法和模型转换代码，必须验证数值精度、溢出、饱和、边界和平台一致性。

### 20.3 最低测试门槛建议

| 变更类型 | 最低要求 |
|---|---|
| 非安全关键普通逻辑 | 编译、静态分析、单元测试、必要集成测试。 |
| 通信/诊断变更 | 单元、集成、协议一致性、异常/NRC、总线超时和回归测试。 |
| 安全相关逻辑 | 静态分析、单元、集成、故障注入、覆盖率、SIL/HIL 和安全评审。 |
| 网络安全相关逻辑 | 静态分析、安全测试、模糊测试、权限/认证测试和安全评审。 |
| Bootloader/OTA | 刷写成功/失败、断电、回滚、版本兼容、签名校验、恢复和 HIL 测试。 |
| NVM/标定变更 | 默认值、范围、CRC、版本迁移、掉电、磨损和恢复测试。 |
| AI生成大段代码 | 至少单元+集成+静态分析；关键逻辑逐行审查；必要时 SIL/HIL 和专项评审。 |

## 二十一、静态分析、覆盖率与工具链

### 21.1 静态分析

- 必须使用项目批准的静态分析工具检查 MISRA/CERT/AUTOSAR 规则、未定义行为、越界、空指针、溢出、死代码、竞态和复杂度。
- 静态分析违规必须分类处理：修复、合理偏差、误报或工具限制；不得无说明批量压制。
- 偏差记录必须包含规则编号、位置、原因、风险、替代控制、审批人和复审日期。
- AI 可以辅助解释静态分析结果，但不得替代责任人判断。

### 21.2 覆盖率

- 必须根据项目安全等级和客户要求确定语句、分支、条件、MC/DC 或其他覆盖率目标。
- 覆盖率豁免必须说明不可达原因、平台限制、风险和替代验证方法。
- 安全相关代码的覆盖率报告必须能追踪到测试用例和需求。
- 覆盖率数据必须来自受控构建和受控测试环境，禁止手工拼接无来源数据。

### 21.3 工具链

- 编译器、链接器、代码生成器、静态分析、单元测试框架、覆盖率工具、模型转换工具和刷写工具必须纳入工具清单。
- 对可能引入或漏检错误的工具，必须评估工具风险、验证策略、合格性资料或客户认可要求。
- 编译选项、链接脚本、启动文件、库版本、生成器版本和工具许可证必须可复现、可审计。
- AI 工具用于正式工作产品生成时，必须说明其用途边界、人工验证方法和不确定性控制。

## 二十二、CI/CD、配置管理与发布

### 22.1 配置管理

- 所有源代码、配置、脚本、ARXML、DBC、ODX、A2L、链接脚本、生成器配置、测试脚本和文档必须进入版本控制。
- 必须建立需求基线、设计基线、代码基线、测试基线、工具链基线、标定基线和发布基线。
- 变更必须有唯一 ID、原因、影响分析、评审、测试证据和发布记录。
- 生成代码必须记录生成器版本、输入配置版本、生成时间和校验信息。

### 22.2 CI/CD

- CI 至少必须执行构建、格式检查、静态分析、单元测试、基础安全扫描、生成代码一致性检查和产物归档。
- 安全相关分支必须有更严格门禁，如规则零新增违规、覆盖率阈值、需求追踪检查和测试证据检查。
- 构建产物必须可追踪源代码提交、配置版本、工具链版本、编译选项、标定版本和生成脚本版本。
- 禁止通过跳过 CI、手工替换产物或本地私有构建交付量产软件。

### 22.3 发布与回滚

- 发布必须记录版本、功能范围、已知问题、风险、兼容性、数据库/诊断/标定/配置变更、刷写策略和回滚方案。
- 发布前必须确认监控、诊断、故障记录、DTC、回滚、售后工具和客户交付物。
- 高风险发布应采用灰度、分批、特性开关、A/B 分区或受控试验车队验证。
- 发布后必须观察关键指标、DTC、日志、OTA 成功率、售后反馈和质量问题。

## 二十三、标定、参数与生产数据

### 23.1 标定参数

- 标定参数必须定义名称、含义、单位、范围、默认值、分辨率、依赖关系、安全影响和适用版本。
- 安全相关参数必须有范围保护、权限控制、变更记录和验证证据。
- A2L、标定数据库、标定文件和软件版本必须一致；标定包必须可追踪。
- 参数越界、缺失、CRC 错误、版本不兼容或 NVM 损坏时必须进入受控默认值或安全状态。

### 23.2 生产与制造数据

- EOL、工厂刷写、序列号、VIN、证书、密钥、传感器校准数据和配置码必须纳入安全和配置管理。
- 生产例程必须限制权限和环境，禁止在售后或普通诊断会话中误触发。
- 工厂数据写入必须具备校验、回读、记录、失败重试和防重复策略。
- 不得将真实 VIN、密钥、证书、生产日志或客户车辆数据输入未经批准的 AI 工具。

## 二十四、日志、监控、可观测性与售后诊断

### 24.1 日志规范

- 日志必须分级，明确开发日志、量产日志、诊断日志、安全日志和隐私数据边界。
- 禁止在日志中输出密钥、令牌、证书、种子密钥算法、完整个人信息或可滥用的安全细节。
- 高频路径日志必须控制频率和开关，避免影响实时性、存储寿命或总线负载。
- 关键故障必须记录足够上下文，如软件版本、故障码、状态机状态、输入摘要、时间戳、复位原因和环境状态。

### 24.2 售后诊断

- 售后诊断必须能够定位版本、配置、DTC、快照数据、扩展数据、刷写记录和关键状态。
- 诊断数据导出必须遵守隐私和客户数据规则；AI 分析日志前必须脱敏。
- 量产问题闭环必须连接缺陷、DTC、日志、车辆条件、复现步骤、修复版本和验证证据。

## 二十五、供应链、开源与第三方软件

- 新增第三方软件、开源库、RTOS 组件、加密库、通信栈、Bootloader、驱动或工具必须评估许可证、维护状态、漏洞、功能安全资料、网络安全资料、兼容性和替代方案。
- 必须维护 SBOM 或等效清单，记录组件名称、版本、来源、许可证、用途、漏洞状态和责任人。
- 第三方二进制库必须保存供应商版本、校验值、编译选项、交付说明和已知限制。
- AI 推荐的包名、库或代码片段不得直接采纳，必须确认真实存在、来源可信、许可证可接受且适用于嵌入式量产环境。
- 供应商交付物必须有验收标准，包括文档、接口、测试报告、静态分析、覆盖率、安全资料、漏洞清单和已知问题。

## 二十六、质量门禁与证据包

### 26.1 质量门禁

| 门禁 | 通过标准 |
|---|---|
| 需求门禁 | 需求清晰、可验证、可追踪，安全/网络安全/资源/诊断/OTA 要求明确。 |
| 安全门禁 | 安全计划、ASIL、软件安全需求、安全机制、故障反应和安全验证策略明确。 |
| 网络安全门禁 | TARA、网络安全需求、攻击面、认证授权、密钥、诊断/OTA 安全和测试策略明确。 |
| 设计门禁 | 架构边界、接口、状态机、资源预算、时序预算、异常处理、降级和回滚明确。 |
| 开发门禁 | 编译通过、无未处理警告、静态分析达标、无敏感信息、无明显坏味道。 |
| AI门禁 | AI 输出已人工理解、验证、测试、记录；高风险输出完成专项评审。 |
| 测试门禁 | 单元/集成/SIL/PIL/HIL 按计划通过，关键路径和异常路径覆盖，结果可追溯。 |
| 安全测试门禁 | 故障注入、诊断、降级、看门狗、复位恢复和覆盖率证据完整。 |
| 网络安全测试门禁 | 权限、认证、模糊测试、漏洞扫描、OTA/诊断安全测试完成且无未处理高风险问题。 |
| 发布门禁 | 发布计划、版本、配置、标定、刷写、监控、回滚、客户交付物和负责人确认。 |

### 26.2 合并前最终检查

- [ ] 需求和验收标准已满足，追踪关系完整。
- [ ] 设计方案和实现一致，接口和状态机已评审。
- [ ] 安全/网络安全影响已评估，必要工作产品已更新。
- [ ] 代码已格式化并通过编译、静态分析和规则检查。
- [ ] 新增/修改逻辑有相应单元、集成、异常和回归测试。
- [ ] 资源影响已评估，包括 CPU、RAM、ROM、栈、NVM、总线负载和实时性。
- [ ] 无硬编码密钥、敏感日志、未授权诊断入口和高危依赖。
- [ ] Bootloader/OTA/诊断/NVM/标定/配置变更已完成专项检查。
- [ ] AI 参与范围已说明，AI 输出已人工验证并记录。
- [ ] PR 说明清晰，评审意见已处理，发布和回滚已考虑。

### 26.3 证据包建议

| 证据类别 | 内容 |
|---|---|
| 需求证据 | 需求列表、追踪矩阵、变更记录、验收标准。 |
| 设计证据 | 架构设计、接口定义、状态机、资源预算、降级策略。 |

## 二十八、结语

AI 可以显著提升嵌入式和汽车电子软件开发效率，但车辆软件的最终质量仍来自清晰的需求、可靠的架构、严格的安全与网络安全工程、可验证的代码、充分的台架和车辆测试、透明的评审以及可持续的量产运维。团队应把 AI 当作高效协作者，而不是最终责任人。
