# 嵌入式软件分层测试

> **适用场景**：底层手写C（GCC平台）+ 应用层MATLAB/Simulink（Embedded Coder）  
> **调度方式**：裸机（主循环 + 定时中断）  
> **跨层接口**：手写RTE层（全局变量 / 函数传参）  
> **工具策略**：开源方案为主  
> **功能安全**：部分产品需适配 ISO 26262（ASIL-oriented）

---

## 一、 总体架构与测试策略

### 1.1 软件架构分层

```
┌─────────────────────────────────────────┐
│  应用层 (Application)                   │
│  MATLAB/Simulink → Embedded Coder       │
│  生成代码: model_step(), model_init()   │
├─────────────────────────────────────────┤
│  RTE 层 (手写)                           │
│  全局结构体 / 函数接口传参                 │
│  职责: 解耦应用层与底层，定义数据契约      │
├─────────────────────────────────────────┤
│  底层 (Basic Software)                   │
│  手写 C + 厂商 SDK (S32K/BYD HAL)        │
│  驱动: ADC, PWM, CAN, GPIO, Diagnostics  │
├─────────────────────────────────────────┤
│  硬件 (MCU)                              │
│  S32K3xx / BYD 系列 / 多平台              │
└─────────────────────────────────────────┘
```

### 1.2 为什么必须分层测试？

| 风险点 | 不分层测试的后果 | 分层测试的收益 |
|--------|------------------|----------------|
| Simulink 生成代码与手写 C 数据类型宽度差异 | 隐式类型转换导致溢出，裸机下无异常捕获直接死机 | SIL 阶段在 PC 端暴露所有数值精度问题 |
| 裸机中断与 Simulink `step()` 时序冲突 | 中断嵌套导致 RTE 全局变量半更新状态被读取 | PIL 阶段在真实时钟下验证原子性 |
| 多芯片 GCC 编译器版本差异 | 同一套底层代码在不同芯片上优化行为不同 | 单元测试在 PC 端验证逻辑，PIL 验证编译器行为 |
| 手写 RTE 全局变量内存对齐 | 结构体打包差异导致 CAN 信号解析错位 | 跨层测试用 `static_assert` 锁定内存布局 |

### 1.3 测试金字塔（V模型映射）

```
          需求分析
             │
    ┌────────┴────────┐
    │                 │
[应用层需求]      [底层需求]
    │                 │
   MIL              单元测试
    │                 │
   SIL              集成测试
    │                 │
   PIL ──────────── 跨层 RTE 测试
    │                 │
   HIL ←──────────── 系统测试
    │                 │
  实车验证 ────────── 验收测试
```

## 三、 应用层 Simulink 测试（MIL → SIL → PIL）

### 3.1 阶段一：MIL（Model-in-the-Loop）

**目的**：在纯仿真环境中验证算法模型是否满足需求，**不涉及任何代码生成**。

**为什么先做 MIL**：如果模型本身逻辑错误，后续 SIL/PIL 只是验证"错误代码是否被正确生成"。

#### 3.1.1 Step 1：模型准备

```matlab
% 在 MATLAB Command Window 执行
model = 'MyController';  % 您的模型名
open_system(model);

% 配置求解器（裸机通常用定步长）
set_param(model, 'SolverType', 'Fixed-step');
set_param(model, 'FixedStep', '0.01');  % 10ms，匹配您的裸机调度周期
set_param(model, 'SystemTargetFile', 'ert.tlc');  % Embedded Coder
```

#### 3.1.2 Step 2：配置 Embedded Coder 生成可重入代码

**为什么需要可重入**：裸机环境下，如果 Simulink 生成的 `step()` 函数在中断和主循环中被重入调用，非可重入代码会导致全局状态混乱。

在模型中操作：
1. **Code Generation → Interface → Code interface packaging**：选择 `Reusable function`
2. **Code Generation → Interface → Multi-instance code**：勾选 `Yes`
3. **Code Generation → Interface → Pass root-level I/O as**：选择 `Individual arguments`（避免全局变量，便于测试）
4. **Code Generation → Interface → Generate reusable code**：确认勾选

**关键配置截图说明**：
```matlab
% 通过脚本自动化配置（推荐纳入版本控制）
set_param(model, 'CodeInterfacePackaging', 'Reusable function');
set_param(model, 'MultiInstanceERTCode', 'on');
set_param(model, 'RootIOFormat', 'Individual arguments');
set_param(model, 'GenerateReusableCode', 'on');
set_param(model, 'InlineParams', 'on');  % 参数内联，减少全局变量
```

**生成的代码形态**：
```c
/* 可重入函数原型：所有状态通过结构体指针传递 */
void MyController_step(RT_MODEL_MyController *const rtM,
                       const real_T rtU_In1,      /* 输入参数 */
                       const real_T rtU_In2,
                       real_T *rtY_Out1);         /* 输出参数 */

void MyController_init(RT_MODEL_MyController *const rtM);
```

**为什么这样更好**：
- 无隐式全局状态，测试时可实例化多个模型对象
- 与手写 RTE 层对接时，可通过函数传参显式控制数据流
- 便于后续 PIL 阶段在目标芯片上多实例测试

#### 3.1.3 Step 3：编写 MIL 测试用例

**为什么用 Simulink Test Manager**：它支持需求链接、信号构造、自动评估准则，是 ISO 26262 认可的工具。

```matlab
% 创建测试文件
testFile = sltest.testmanager.createTestFile('MyController_MIL.mldatx');

% 创建测试套件
suite = testFile.createTestSuite('Functional_Tests');

% 创建测试用例：正常工况
tc1 = suite.createTestCase('Normal_Operation');
tc1.setProperty('Model', model);

% 设置输入信号（从 Excel/MAT 文件加载或手动构造）
tc1.setProperty('InputLoadingMode', 'InSignalBuilder');
% 或链接到外部数据文件
tc1.setProperty('ExternalInput', 'test_vectors_normal.mat');

% 添加评估准则（Assertion）
assess = tc1.addAssessment('Y_Out1_Range');
assess.setProperty('Type', 'Custom');
assess.setProperty('Expression', 'Y_Out1 >= 0 && Y_Out1 <= 100');
```

#### 3.1.4 Step 4：执行 MIL 并分析覆盖率

```matlab
% 配置覆盖率
coverageSettings = sltest.testmanager.getTestFileCoverageSettings(testFile);
coverageSettings.MetricSetting = 'DecisionCoverage, ConditionCoverage, MCDCCoverage';

% 运行测试
result = testFile.run;

% 查看报告
sltest.testmanager.report(result, 'MIL_Report.pdf', ...
    'IncludeTestResults', true, ...
    'IncludeCoverageResults', true, ...
    'IncludeComparisonSignalPlots', true);
```

**覆盖率目标**：
- 非功能安全产品：Decision Coverage ≥ 90%
- 功能安全产品（ASIL C/D）：MC/DC Coverage ≥ 100%

### 3.2 阶段二：SIL（Software-in-the-Loop）

**目的**：验证 Embedded Coder 生成的 C 代码与 Simulink 模型在数值上**完全等价**（背靠背测试）。

**为什么 SIL 是性价比最高的环节**：它在 PC 上运行生成代码，无需目标硬件，能发现 90% 的代码生成问题（如定点溢出、除零保护缺失、优化导致的数值差异）。

#### 3.2.1 Step 1：配置 SIL 模式

```matlab
% 在 Test Manager 中，将测试用例的执行模式改为 SIL
set_param(model, 'SimulationMode', 'software-in-the-loop (SIL)');

% 或针对单个测试用例
tc1.setProperty('SimulationMode', 'SIL');
```

#### 3.2.2 Step 2：处理多平台代码差异

**为什么需要关注**：Embedded Coder 生成的代码可能在不同 GCC 版本下产生不同的浮点精度行为。

```matlab
% 在 SIL 配置中指定编译器（使用主机 GCC 模拟）
set_param(model, 'SimCompilerOptimizationLevel', 'Off');  % 关闭优化，便于调试
set_param(model, 'SimBuildMode', 'Model reference');
```

#### 3.2.3 Step 3：背靠背比较（B2B）

```matlab
% 先执行 MIL 保存基准结果
result_MIL = tc1.run('SimulationMode', 'normal');
baseline = result_MIL.getOutputData;

% 再执行 SIL
result_SIL = tc1.run('SimulationMode', 'SIL');
sil_output = result_SIL.getOutputData;

% 自动比较（容差设置）
tolerance = 1e-6;  % 浮点容差
diff = abs(baseline - sil_output);
assert(all(diff < tolerance), 'SIL output deviates from MIL baseline!');
```

**常见 SIL 失败原因与对策**：

| 失败现象 | 根因 | 对策 |
|----------|------|------|
| 输出存在固定偏差 | 代码生成时浮点精度降级（double→float） | 模型中显式指定数据类型为 `single`；或在配置中禁用降级 |
| 特定输入下输出跳变 | 代码优化导致运算顺序改变 | `set_param(model, 'CompilerOptimizationLevel', 'Off')` |
| 饱和逻辑差异 | 模型中 Saturation 块与生成代码边界检查顺序不同 | 检查模块参数 "Saturate on integer overflow" |

### 3.3 阶段三：PIL（Processor-in-the-Loop）

**目的**：在**真实目标芯片**上运行生成的代码，验证编译器优化、内存对齐、执行时间、栈使用。

**为什么 PIL 不能跳过**：GCC ARM 的优化级别 `-O2` 可能与 PC 端 MSVC/GCC x86 产生不同的代码行为，特别是位操作和 volatile 访问。

#### 3.3.1 Step 1：配置 PIL 连接器

```matlab
% 针对 S32K3xx 配置（需安装对应支持包）
set_param(model, 'SimulationMode', 'processor-in-the-loop (PIL)');
set_param(model, 'PILConnectivityConfig', 'serial');  % 或 'custom'

% 自定义 PIL 通信（通过 OpenOCD + GDB Server）
set_param(model, 'PILCommunicationInterface', 'Custom');
```

**为什么用 Custom PIL**：S32K/BYD 等芯片可能无官方 PIL 支持包，需通过 GDB 手动实现代码下载和结果回传。

#### 3.3.2 Step 2：手动 PIL 流程（当无官方支持包时）

若 MATLAB 无直接支持包，采用**半自动化 PIL**：

1. **生成代码**：
```matlab
slbuild(model);  % 生成 ert 代码到 slprj/ert/_sharedutils/
```

2. **创建 PIL 工程**：将生成代码 + RTE 接口层 + 厂商启动文件集成到一个裸机工程

3. **添加计时与回传代码**：
```c
/* main_pil.c - 目标芯片侧 */
#include "MyController.h"
#include "timer.h"  /* 厂商 SDK 的定时器 */

RT_MODEL_MyController rtM;
real_T input1, input2, output1;
uint32_t cycleCount;

int main(void) {
    BOARD_Init();       /* 厂商 SDK 初始化 */
    TIMER_Init();       /* 初始化用于计时的定时器 */
    MyController_init(&rtM);

    while (1) {
        /* 从调试通道接收输入（如通过 UART/SWO） */
        input1 = PIL_ReceiveDouble();
        input2 = PIL_ReceiveDouble();

        /* 测量执行周期 */
        TIMER_Reset();
        MyController_step(&rtM, input1, input2, &output1);
        cycleCount = TIMER_GetCycles();

        /* 回传输出和执行时间 */
        PIL_SendDouble(output1);
        PIL_SendUInt32(cycleCount);
    }
}
```

4. **PC 端测试脚本**：
```python
# pil_host.py
import serial
import struct
import numpy as np

ser = serial.Serial('/dev/ttyUSB0', 115200, timeout=1)

# 加载 MIL 测试向量
test_vectors = np.loadtxt('test_vectors.csv', delimiter=',')
mil_outputs = test_vectors[:, 2]  # 假设第3列为 MIL 期望输出

for i, (in1, in2, expected) in enumerate(test_vectors):
    ser.write(struct.pack('<ddd', in1, in2, 0))  # 发送输入
    out = struct.unpack('<d', ser.read(8))[0]     # 接收输出
    cycles = struct.unpack('<I', ser.read(4))[0]  # 接收周期数

    assert abs(out - expected) < 1e-5, f"PIL mismatch at step {i}"
    assert cycles < MAX_ALLOWABLE_CYCLES, f"Timing violation at step {i}"
```

#### 3.3.3 Step 3：PIL 验证清单

| 验证项 | 通过标准 | 为什么重要 |
|--------|----------|-----------|
| 数值等价性 | PIL 输出与 SIL 输出偏差 < 1e-5 | 确认编译器优化未改变算法语义 |
| 执行时间 | `step()` 周期 < 调度周期 × 80% | 裸机下若超时会导致控制周期抖动 |
| 栈使用 | 最大栈深度 < 芯片 SRAM 的 50% | 预留中断嵌套和 RTE 全局变量空间 |
| 代码体积 | `.text` 段 < Flash 容量的 70% | 预留 Bootloader、诊断、标定空间 |

## 四、 底层手写 C 代码测试（单元 → 集成）

### 4.1 测试策略设计

**为什么底层要独立于硬件测试**：厂商 SDK 的寄存器操作在 PC 上无法直接执行，必须通过 Mock 打桩隔离。

```
底层代码结构示例：
├─ driver/           # 厂商 SDK（不测试，假设已验证）
├─ hal/              # 硬件抽象层（Mock 测试）
│   ├─ adc_hal.c     # 封装 SDK 的 ADC 接口
│   ├─ can_hal.c
│   └─ dio_hal.c
├─ services/         # 业务逻辑层（重点测试）
│   ├─ diag_service.c    # 诊断服务
│   ├─ filter_service.c  # 信号滤波
│   └─ safety_service.c  # 安全监控
└─ rte/              # RTE 层（跨层接口）
    └─ rte_signals.c
```

### 4.2 单元测试环境搭建

#### 4.2.1 Step 1：创建测试工程结构

```
project/

│   ├── hal/
│   ├── services/
│   └── rte/
```

#### 4.2.2 Step 2：配置 CMock（自动 Mock 生成）

**为什么用 CMock 而非手写桩**：手写桩在接口变更时需要同步修改，CMock 通过解析头文件自动生成，维护成本低。

创建 `tests/cmock_config.yml`：
```yaml
:cmock:
  :plugins:
    - ignore
    - callback
    - return_thru_ptr
  :treat_externs: :include  # Mock 外部函数
  :mock_path: tests/mocks
  :verbosity: 2
```

生成 Mock：
```bash
cd tests
ruby /opt/cmock/lib/cmock.rb -o cmock_config.yml ../src/hal/adc_hal.h
# 生成 tests/mocks/Mockadc_hal.c 和 .h
```

#### 4.2.3 Step 3：编写单元测试（以诊断服务为例）

```c
/* tests/test_services/test_diag_service.c */
#include "unity.h"
#include "diag_service.h"
#include "Mockadc_hal.h"      /* CMock 生成 */
#include "Mockcan_hal.h"
#include "Mocktimer_hal.h"

/* 在每个测试前执行 */
void setUp(void) {
    diag_init();
}

void tearDown(void) {
    /* 清理全局状态 */
}

/* 测试用例 1：正常电压范围内的诊断 */
void test_Diag_VoltageNormal_NoFault(void) {
    /* Arrange: 配置 Mock */
    ADC_Value_t mock_adc = { .raw = 2048, .status = ADC_OK };
    adc_read_ExpectAndReturn(ADC_CHANNEL_1, &mock_adc, ADC_OK);

    /* Act */
    DiagResult_t result = diag_check_voltage(ADC_CHANNEL_1);

    /* Assert */
    TEST_ASSERT_EQUAL(DIAG_PASS, result);
}

/* 测试用例 2：电压超上限故障 */
void test_Diag_VoltageOverRange_SetFault(void) {
    ADC_Value_t mock_adc = { .raw = 4095, .status = ADC_OK };
    adc_read_ExpectAndReturn(ADC_CHANNEL_1, &mock_adc, ADC_OK);

    /* 期望 CAN 发送故障帧 */
    CAN_Frame_t expected_frame = { .id = 0x123, .dlc = 8 };
    can_send_ExpectAndReturn(&expected_frame, CAN_OK);

    DiagResult_t result = diag_check_voltage(ADC_CHANNEL_1);
    TEST_ASSERT_EQUAL(DIAG_FAIL_OVERVOLTAGE, result);
}

/* 测试用例 3：ADC 读取超时 */
void test_Diag_AdcTimeout_Fallback(void) {
    ADC_Value_t mock_adc;
    adc_read_ExpectAndReturn(ADC_CHANNEL_1, &mock_adc, ADC_TIMEOUT);

    /* 期望使用上一次有效值，不触发新故障 */
    DiagResult_t result = diag_check_voltage(ADC_CHANNEL_1);
    TEST_ASSERT_EQUAL(DIAG_PASS, result);  /* 降级策略：保持上一次状态 */
}
```

## 五、 跨层 RTE 接口集成测试

### 5.1 为什么 RTE 是最高风险点

您的 RTE 层使用**全局变量传参**，这在裸机中是最常见的做法，但存在以下风险：

| 风险 | 后果 | 检测方法 |
|------|------|----------|
| 结构体成员对齐差异 | Simulink 生成代码与手写 C 对同一结构体 sizeof 不同 | `static_assert(sizeof(RTE_Signal_t) == EXPECTED_SIZE)` |
| 数据类型宽度差异 | `int` 在 Simulink 为 32bit，在 BYD 芯片可能 16bit | 显式使用 `stdint.h`（`int32_t`） |
| 全局变量非原子访问 | 中断中修改 RTE 变量，主循环读取到半更新值 | 关中断保护测试 + PIL 验证 |
| 初始化时序 | Simulink `init()` 与底层 `init()` 调用顺序错误 | 跨层集成测试用例 |

### 5.2 RTE 接口契约定义

**为什么必须先定义契约**：契约是 Simulink 与手写 C 之间的"法律文件"，双方各自基于此开发，最后对接。

```c
/* rte_interface.h - 双方共同遵守 */
#ifndef RTE_INTERFACE_H
#define RTE_INTERFACE_H

#include <stdint.h>
#include <stdbool.h>

/* 显式指定宽度和对齐 */
#pragma pack(push, 1)  /* 关键：确保 Simulink 与 GCC 对齐一致 */

typedef struct {
    int16_t  motor_torque;      /* Q8.8 定点格式 */
    uint16_t motor_speed_rpm;
    uint8_t  fault_code;        /* 0=OK, 1=OverTemp, 2=OverCurrent */
    bool     enable_flag;
    uint8_t  reserved[1];       /* 填充到 8 字节对齐 */
} RTE_MotorOut_t;               /* 期望 sizeof = 8 */

#pragma pack(pop)

/* 编译时断言检查大小和对齐 */
_Static_assert(sizeof(RTE_MotorOut_t) == 8, "RTE_MotorOut_t size mismatch!");
_Static_assert(_Alignof(RTE_MotorOut_t) == 1, "RTE_MotorOut_t alignment error!");

/* 接口函数（显式传参，避免全局变量隐式依赖） */
void RTE_Write_MotorOut(const RTE_MotorOut_t *sig);
const RTE_MotorOut_t* RTE_Read_MotorOut(void);

#endif
```

## 六、 系统级 HIL 测试

### 6.1 HIL 测试范围

**为什么 HIL 不可替代**：前面的测试都是"部分真实"（SIL 在 PC，PIL 只有算法在真实芯片），HIL 是**全软件栈 + 真实 ECU + 仿真环境**。

```
HIL 环境架构：
┌─────────────┐         ┌─────────────┐         ┌─────────────┐
│   PC 仿真机  │ ← CAN → │   目标 ECU  │ ← GPIO → │  负载仿真   │
│ (Python/    │         │ (S32K/BYD)  │         │ (电阻/电机) │
│  Simulink)  │         │ 全软件栈    │         │             │
└─────────────┘         └─────────────┘         └─────────────┘
```

### 6.2 开源 HIL 方案

由于预算限制，推荐**轻量级 HIL**：

| 组件 | 开源方案 | 作用 |
|------|----------|------|
| 仿真主机 | Python + `cantools` + `python-can` | 发送/接收 CAN 报文，模拟传感器 |
| 时序控制 | Python `asyncio` 或 `threading.Timer` | 精确控制 10ms/100ms 周期 |
| 故障注入 | 继电器模块 + GPIO 控制 | 模拟短路、断路 |
| 数据记录 | `pandas` + `matplotlib` | 自动对比预期与实际输出 |

### 6.3 HIL 测试步骤

#### 6.3.1 Step 1：搭建通信桥

```python
# hil_host.py
import can
import cantools
import pandas as pd
import time

# 加载 DBC 文件（描述 CAN 信号）
db = cantools.database.load_file('my_vehicle.dbc')

# 连接 CAN 接口（USB-CAN 适配器，如 CANable/CandleLight）
bus = can.interface.Bus(channel='can0', bustype='socketcan')

# 测试用例：电机启动序列
def test_motor_startup():
    results = []

    # T=0: 发送上电信号
    msg_power = db.get_message_by_name('VCU_Power')
    bus.send(can.Message(
        arbitration_id=msg_power.frame_id,
        data=msg_power.encode({'SystemPower': 1})
    ))

    # T=100ms: 发送转速请求
    time.sleep(0.1)
    msg_req = db.get_message_by_name('VCU_MotorReq')
    bus.send(can.Message(
        arbitration_id=msg_req.frame_id,
        data=msg_req.encode({'TargetSpeed': 1000, 'Enable': 1})
    ))

    # 监听响应（超时 500ms）
    start_time = time.time()
    while time.time() - start_time < 0.5:
        msg = bus.recv(timeout=0.01)
        if msg and msg.arbitration_id == db.get_message_by_name('MCU_Status').frame_id:
            decoded = db.decode_message(msg.arbitration_id, msg.data)
            results.append({
                'time': time.time() - start_time,
                'actual_speed': decoded['ActualSpeed'],
                'fault': decoded['FaultCode']
            })

    # 验证：100ms 内电机应响应，且无故障
    df = pd.DataFrame(results)
    assert len(df) > 0, "No response from ECU"
    assert df['fault'].iloc[0] == 0, f"Fault detected: {df['fault'].iloc[0]}"
    assert df['actual_speed'].iloc[-1] > 0, "Motor did not start"

    return df

if __name__ == '__main__':
    df = test_motor_startup()
    df.to_csv('hil_result_motor_startup.csv')
    print("HIL test passed!")
```

#### 8.2.1 MC/DC 覆盖率（针对 Simulink 生成代码）

#### 8.2.2 底层 C 代码 MC/DC（开源替代方案）

由于 `gcov` 原生不支持 MC/DC，需手动插桩：

```c
/* 原始代码 */
if ((speed > MAX_SPEED) && (temp > MAX_TEMP) && (enable == true)) {
    shutdown_motor();
}

/* 插桩后（用于 MC/DC 验证） */
bool cond_a = (speed > MAX_SPEED);
bool cond_b = (temp > MAX_TEMP);
bool cond_c = (enable == true);

/* 记录条件向量到测试缓冲区 */
TEST_RECORD_MCDC(cond_a, cond_b, cond_c, (cond_a && cond_b && cond_c));

if (cond_a && cond_b && cond_c) {
    shutdown_motor();
}
```

测试后分析缓冲区，验证是否覆盖：
- (T, X, X) → True
- (F, T, T) → False
- (T, F, T) → False
- (T, T, F) → False

### 8.3 工具鉴定（Tool Qualification）

**为什么**：ISO 26262 要求用于安全相关开发的工具必须经过鉴定，证明其不会引入错误。

| 工具 | 用途 | ASIL 等级 | 鉴定方法 |
|------|------|-----------|----------|
| GCC ARM | 编译 | ASIL D | 使用已鉴定版本（如 GCC 10.3.1 经 TÜV 认证） |
| Simulink Test | 测试执行 | ASIL D | MathWorks 提供 TÜV 证书 |
| Unity/CMock | 单元测试 | ASIL B/C | 自行鉴定：分析源码 + 验证用例 |
| OpenOCD | 调试下载 | ASIL A/B | 仅用于开发，不用于最终验证可免鉴定 |

**开源工具鉴定步骤**：
1. **工具分类**：确定 TCL（Tool Confidence Level）
2. **风险分析**：分析工具可能引入的错误（如 CMock 生成的桩函数签名错误）
3. **预防措施**：
   - 锁定工具版本（`git checkout` 固定 tag）
   - 编写"工具验证用例"（如验证 CMock 是否正确解析函数指针）
4. **文档输出**：《工具鉴定报告》 + 《工具验证记录》

| SYS-001 | 电机超速保护 | Simulink: `OverspeedProtection` 模块 | MIL-001, SIL-001, PIL-001 | MIL/SIL/PIL | 100% Decision | Pass |
| SYS-002 | 温度传感器故障诊断 | C: `diag_check_temp()` | UNIT-005, INT-003 | Unit/Integration | 100% Branch | Pass |

## 十、 附录 A：多芯片 GCC 适配说明

### 10.1 S32K3xx (Cortex-M7)

```bash
# 编译器标志
CFLAGS += -mcpu=cortex-m7 -mthumb -mfpu=fpv5-d16 -mfloat-abi=hard
CFLAGS += -O2 -g3
CFLAGS += -fsingle-precision-constant  # 强制单精度浮点常数
```

### 10.2 BYD 芯片（假设 Cortex-M/R）

```bash
# 通常使用标准 ARM GCC，但需确认厂商是否修改了启动文件和链接脚本
CFLAGS += -mcpu=cortex-m4 -mthumb -mfpu=fpv4-sp-d16 -mfloat-abi=hard
# 注意：BYD 芯片可能使用自定义指令集扩展，需在 PIL 阶段验证
```

### 10.3 跨平台一致性保证

```cmake
# CMakeLists.txt 中根据芯片选择工具链
if(TARGET_CHIP STREQUAL "S32K3")
    set(CMAKE_C_COMPILER arm-none-eabi-gcc)
    set(CMAKE_SYSTEM_PROCESSOR cortex-m7)
elseif(TARGET_CHIP STREQUAL "BYD_XXX")
    set(CMAKE_C_COMPILER arm-none-eabi-gcc)
    set(CMAKE_SYSTEM_PROCESSOR cortex-m4)
endif()
```

## 十一、 附录 B：常见错误与排查

| 错误现象 | 可能原因 | 排查方法 |
|----------|----------|----------|
| SIL 与 MIL 输出偏差大 | 数据类型不匹配（模型用 double，代码生成用 float） | 检查模型 Configuration → Hardware Implementation → 设备类型 |
| PIL 链接失败 | 启动文件（.s）与芯片不匹配 | 确认使用厂商 SDK 提供的正确启动文件 |
| CMock 生成编译错误 | 头文件包含复杂宏或内联汇编 | 在 `cmock_config.yml` 中添加 `:strippables` 过滤 |
| 覆盖率数据缺失 | 编译时未加 `--coverage` | 检查 CMake 中 `CMAKE_C_FLAGS` 是否包含 `-fprofile-arcs` |
| RTE 结构体大小不一致 | `#pragma pack` 或位域差异 | 在两边代码中同时打印 `sizeof` 和 `offsetof` 对比 |
| 单元测试通过但集成失败 | Mock 返回值设置过于理想，未覆盖真实边界 | 在集成测试中使用真实 HAL 或更保守的 Mock 值 |
